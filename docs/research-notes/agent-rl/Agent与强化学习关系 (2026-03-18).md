# 对话整理版：用强化学习视角做出更好的 Agent 系统

**整理日期：** 2026-05-20  
**原始对话日期：** 2026-03-18  
**项目定位：** 本项目不是为了做一个封闭的 RL demo，也不是为了把 Agent 简化成“会调用工具的 LLM”。它的目标是：借助强化学习的状态、动作、奖励、策略、轨迹、环境、POMDP/SMDP 等思想，更系统地理解并实现一个可观察、可评估、可扩展、可持续改进的 Agent 系统。

---

## 0. 项目核心观念

### 0.1 Agent 是系统，不是单个模型

一个现代 Agent 应该被看作运行系统：

```text
Agent = LLM/Policy + State/Memory + Tools + Planner + Runtime Loop + Constraints + Evaluation
```

LLM 只是其中的策略实现器、规划器、压缩器、评估器或世界模型之一，不等于整个 Agent。

### 0.2 强化学习是理解和优化 Agent 的方法论

强化学习提供了一套非常好的分析语言：

- `state / observation`：Agent 当前知道什么。
- `action`：Agent 能做什么，包括工具调用、检索、写文件、调用子 Agent、请求人工审批。
- `policy`：Agent 如何选择下一步。
- `reward / evaluator`：什么结果更好。
- `trajectory`：一次任务完整执行过程。
- `environment`：用户、代码仓库、浏览器、数据库、工具和其他 Agent 组成的外部世界。

但本项目的重点不是一开始训练一个策略网络，而是先把 Agent runtime 做成可运行、可记录、可评估、可替换策略的系统。RL 后续用于优化局部决策器，例如工具选择、检索策略、停止策略、反思策略、子 Agent 路由、预算分配。

### 0.3 真实 LLM Agent 更像带记忆和工具 options 的 POMDP/SMDP

真实 Agent 拿不到完整环境状态，只能看到局部观测；它还要维护内部记忆、调用持续时间不等的工具、等待审批、调度子 Agent。因此比普通 MDP 更贴近：

```text
memory-augmented constrained hierarchical semi-Markov POMDP
```

对应到工程实现，就是：

- 记忆不是普通聊天历史，而是 belief state 的工程近似。
- 工具调用、子 Agent、审批流程是 macro-action / option。
- guardrails、权限、成本、延迟、安全是约束，不只是 reward。
- runtime 要记录轨迹，否则无法复盘、评估和学习。

## 0.4 当前先进 Agent 系统要重点学习的模块

### A. Runtime 与编排

学习目标：理解 Agent 如何稳定执行多步任务。

重点概念：

- ReAct：每一步根据最新观测重新决策。
- Plan-and-Execute：先计划，再执行，适合长任务。
- Multi-Agent：多个局部策略或角色协作。
- Handoff：把任务委托给更专门的 Agent。
- Durable execution：长任务中断后可恢复。

可参考实现：

- OpenAI Agents SDK：agents、tools、handoffs、guardrails、sessions、tracing。
- LangGraph：stateful、long-running、human-in-the-loop agent orchestration。
- AutoGen / CrewAI：多 Agent 角色协作与流程编排。

### B. RAG 与知识系统

学习目标：让 Agent 能在外部知识库、项目文档、代码仓库和历史材料中找到可靠上下文。

重点概念：

- query rewriting / decomposition
- hybrid retrieval：向量检索 + 关键词检索
- reranking
- chunking strategy
- citation / grounding
- retrieval evaluation
- agentic RAG：Agent 自己决定何时检索、检索什么、如何验证检索结果

RAG 不是“把所有资料塞进向量库”。对 Agent 来说，RAG 是 observation builder：它决定 Agent 每一步看见什么。

### C. 记忆与记忆压缩

学习目标：让 Agent 跨任务积累经验，同时避免上下文污染。

记忆至少分四类：

- Working memory：当前任务的短期状态。
- Episodic memory：历史任务轨迹和经验。
- Semantic memory：稳定知识、用户偏好、项目事实。
- Procedural memory：可复用工作流、技能、策略。

关键机制：

- memory write policy：什么时候写入记忆。
- memory retrieval policy：什么时候检索记忆。
- memory compression：长上下文摘要、结构化提炼、冲突合并。
- memory validation：防止错误记忆污染后续决策。
- forgetting / decay：过期信息要降权或淘汰。

可参考实现：

- mem0：面向 AI assistant/agent 的持久化上下文记忆。
- Letta / MemGPT：让 Agent 通过工具调用自主管理长期记忆和上下文窗口。
- LangGraph / LangChain memory：短期 thread state、长期 store、checkpointer。

### D. 评估、奖励与 RL

学习目标：用强化学习思想让 Agent 可优化。

短期先做：

- trajectory logging
- task success evaluator
- cost / latency / safety metrics
- regression tasks
- prompt / policy A-B test

中期再做：

- 从轨迹中提取偏好数据。
- 训练或选择小策略：工具选择、检索、停止、反思、路由。
- bandit / offline RL / preference optimization。

长期才考虑：

- PPO / GRPO / RLHF / RLAIF 等更重的策略优化。
- 多 Agent 协作中的 credit assignment。

### E. 安全、约束与可观测性

学习目标：让 Agent 在真实环境中可控。

重点概念：

- guardrails
- permission boundary
- tool call validation
- sandbox
- approval gate
- tracing
- replay
- cost budget
- failure recovery

一个 Agent 系统是否成熟，不看它能不能单次炫技，而看它能不能在错误、噪声、权限、成本、长任务和不完整信息下稳定工作。

# 一、先给结论：它们的关系到底是什么

**Date:** 2026-03-18
**Source:** [chat.openai.com](https://chatgpt.com/c/69ba3a1a-0d88-832e-86d0-7a4097801406)

---

### **You**

从工程角度，Agent 不是一个模型名字，也不是“会调用工具的 LLM”这么简单。
更准确地说，Agent 是一个带有目标、状态、动作空间和控制循环的运行系统。

一个最小的 Agent，通常至少有这 5 个部分：

Goal / Instruction（目标）
它到底要完成什么。
例如：“帮我整理客户邮件并更新 CRM”。

State（状态）
它当前知道什么，执行到哪一步。
比如：用户历史、工具调用结果、任务进度、记忆。

Action Space（动作空间）
它能做什么。
比如：搜索网页、查数据库、发邮件、调用子 Agent、请求人工审批。

Policy / Planner（策略或规划）
它下一步怎么决定。
这一步可以是 LLM 推理，也可以是代码里的固定逻辑，也可以是两者混合。

Loop / Runtime（执行循环）
它如何持续运行直到结束。
即：读输入 → 决策 → 调工具 → 更新状态 → 再决策 → 结束。

所以从系统设计上，Agent 其实是：

LLM + 状态管理 + 工具系统 + 决策循环 + 运行时约束   这个Agent 的理论，和强化学习的理论有什么关系，我感觉很相似，详细剖析一下

---

### **ChatGPT**

你的直觉是对的：Agent 理论和强化学习（RL）在结构上非常相似，但它们并不等价。
更准确地说：
Agent 是一个更大的系统概念；强化学习是其中一种“让 Agent 学会决策”的方法。
也可以反过来说：

Agent 关心的是：这个系统如何感知、记忆、决策、行动，并在环境中持续运行。

强化学习 关心的是：如果一个 Agent 反复和环境交互，如何根据奖励信号学出一个更好的策略。

所以两者像是：

Agent = “完整运行体”

RL = “训练这个运行体中决策部分的一类理论与方法”

下面我按工程和理论两条线，把它们拆开讲透。

一、先给结论：它们的关系到底是什么
你给的 Agent 定义：

Goal + State + Action Space + Policy/Planner + Loop/Runtime

如果把它映射到强化学习，几乎可以一一对应：

Goal → 奖励函数 / 任务目标

State → 马尔可夫状态 / 历史观测 / belief state

Action Space → 动作集合

Policy / Planner → 策略 π(a|s) / 价值函数 / 规划器

Loop / Runtime → 环境交互过程

所以你会觉得像，是因为它们共享同一个抽象骨架：

“在环境中，根据状态选择动作，以实现目标。”

这是人工智能里非常底层的统一视角。
但关键区别在于：
RL 默认研究的是“学习最优决策”
而今天工程上说的 Agent，很多时候研究的是“把一个能工作的自主系统搭起来”。
这两者关注点不同：
强化学习更偏理论
它问的是：

什么是状态？

什么是最优策略？

如何最大化长期回报？

探索和利用如何平衡？

在部分可观测环境中怎么办？

学习是否收敛？

Agent 工程更偏系统
它问的是：

如何把用户目标分解成子任务？

怎么调用外部工具？

记忆怎么存？

错误怎么恢复？

什么时候该停？

什么时候要人工审批？

如何约束成本、延迟、权限和安全？

所以：
RL 是 Agent 决策问题的数学化；Agent 工程是把决策、工具、状态、约束和运行时整合成可用系统。

二、从经典 AI 角度看：Agent 概念其实比 RL 更大
在 AI 教科书里，Agent 的定义很老：

Agent 是一个通过传感器感知环境、通过执行器作用于环境的实体。

这是最一般的定义。
在这个定义下，下面都叫 Agent：

恒温器

自动驾驶系统

游戏 AI

机器人

LLM Agent

会自动发邮件和更新 CRM 的工作流系统

而强化学习只是研究其中一类 Agent：

当环境反馈以奖励形式给出，且 Agent 需要通过试错学习策略时，如何学？

所以历史顺序上看，不是“Agent 来自 RL”，而更像是：

Agent 是总概念

RL 是研究 Agent 行为优化的一条主线

LLM Agent 是近几年把语言模型作为策略模块/世界模型/规划器后形成的新工程形态

三、为什么你会觉得“几乎一样”——因为底层形式真的高度同构
我们把两者做严格映射。

1. Goal vs Reward
在工程 Agent 里，目标可能是：

“整理客户邮件并更新 CRM”

“帮我订一张下周去上海的机票”

“分析报表并输出摘要”

这些目标往往是自然语言目标，是高层语义描述。
在 RL 里，目标通常被转换成：

每一步奖励 r_t

或累计回报 Gt=∑γkrt+kG_t = \\sum \\gamma^k r_{t+k}Gt​=∑γkrt+k​

也就是说：

Agent 工程里的 Goal，通常是人类可读的任务规范

RL 里的 Goal，通常是数学可优化的奖励定义

这是第一层关键差异。
工程难点
很多真实世界任务很难写成好的 reward function。
比如“帮我写一封得体、专业、不冒犯人的邮件”，你很难定义一个精确奖励。
所以 LLM Agent 里大量目标不是“奖励函数”，而是：

prompt instruction

policy constraints

审批规则

检查器

rubric

测试用例

这说明：
现代 Agent 往往不是通过显式 reward 来驱动，而是通过“语言目标 + 规则 + 反馈”驱动。
这和标准 RL 已经拉开距离了。

2. State vs Observation / Belief
你说的 State 包含：

用户历史

工具调用结果

任务进度

记忆

这和 RL 的状态概念也很接近。
在 RL 里：

如果环境完全可观测，就有状态 sts_tst​

如果不完全可观测，就只有 observation oto_tot​，需要构造 belief state 或 history

而真实 Agent 几乎总是部分可观测的：

用户真实意图不完全可见

外部世界变化不完全可见

工具返回可能不完整

历史上下文有限

长期记忆可能是近似的、检索式的

所以一个 LLM Agent 的“状态”往往不是 RL 那种干净的数学状态，而是工程拼接态：

当前会话上下文

检索到的记忆

tool outputs

scratchpad / plan

当前步骤编号

pending actions

审批标志

budget / token / latency constraints

因此它更像：

POMDP（部分可观测马尔可夫决策过程）+ 外挂记忆系统

这一点和 RL 很像，但工程形态更脏、更异构。

3. Action Space 完全对应
这部分最像。
RL 里的动作空间：

离散动作：上/下/左/右

连续动作：转角、力度、速度

Agent 里的动作空间：

搜索网页

调数据库

写文件

发邮件

调子 Agent

请求人工确认

结束任务

本质上，都是：

在当前状态下可执行的控制操作集合

只是现代 Agent 的动作不再是“物理动作”，而是API 动作、符号动作、工作流动作。
所以 LLM Agent 可以看成一种高层离散动作决策系统。
甚至 function calling 本身就可以被理解为：

action schema

parameterized action

action execution result → next observation

这和 RL 里 parameterized action 很接近。

4. Policy / Planner 是最核心的相似处
在 RL 中，策略是：
π(a∣s)\\pi(a \\mid s)π(a∣s)
即：在状态 s 下如何选动作 a。
在 Agent 里，Policy / Planner 也是这个问题：

当前该搜网页还是发邮件？

先调用 CRM 还是先总结邮件？

要不要请求用户确认？

任务要不要拆成三步？

失败后要不要重试？

差别在于：
RL 里的 policy
通常是训练出来的函数：

Q-learning

policy gradient

actor-critic

model-based planning

LLM Agent 里的 policy
往往是混合体：

一部分由 LLM 推理

一部分由系统 prompt 规定

一部分由代码写死

一部分由规则引擎控制

一部分由人工审批门控

所以现代 Agent 的策略更像：

learned policy + symbolic planner + hard constraints + runtime heuristics

这比经典 RL 的纯策略网络复杂得多。

5. Loop / Runtime vs RL interaction loop
RL 的基本循环：

观察状态

选择动作

环境反馈

得到奖励

更新策略

继续

Agent 的基本循环：

读取输入和当前状态

决策下一步

调用工具

获取结果

更新状态

决定继续/结束

你会发现二者几乎就是一个模板。
区别是：
RL 循环的重点
在“学习”
也就是每轮交互后如何更新参数。
Agent runtime 的重点
在“执行”
也就是如何安全、稳定、低成本地把任务做完。
很多 LLM Agent 在运行时根本不在线学习。
它只是执行时循环，而不改模型参数。
所以：

有循环 ≠ 强化学习

会选动作 ≠ 强化学习

会调工具 ≠ 强化学习

只有当它通过环境反馈来优化策略时，才真正进入 RL 范畴。

四、最关键的区别：Agent 不一定“学”，RL 必须“学”
这是最该抓住的一点。
一个现实中的 LLM Agent 系统可以这样工作：

模型固定

prompt 固定

tool set 固定

runtime 维护状态

根据当前上下文决定调用哪个工具

执行到结束

这已经是 Agent 了。
但它并没有进行 RL 意义上的学习。
也就是说：
Agent 是运行时概念
强调 autonomous execution。
RL 是训练时概念
强调 policy optimization。
这两者的“阶段”都不一样。
你可以有：
1. 不用 RL 的 Agent
最常见

LLM + tool calling + memory + planner

完全靠 prompt 和规则工作

没有任何 reward-based learning

2. 用 RL 训练过某些模块的 Agent
比如

模型经过 RLHF / RLAIF

工具选择策略经过 bandit 或 RL 优化

多步任务完成率通过 RL 调优

3. 真正在线学习的 Agent
更少见，更难部署

根据长期反馈更新策略

在生产中逐步适应用户偏好

面临安全、分布漂移、灾难性遗忘等问题

所以不能把 Agent 和 RL 画等号，但它们有天然亲缘关系。

五、如果用 MDP/POMDP 语言重述 Agent，会非常顺
这是理解两者关系最有效的办法。
一个工程 Agent 往往可以抽象成 POMDP：

Hidden state：真实世界状态，用户真实需求，外部系统真实数据

Observation：用户消息、检索结果、API 返回、网页内容

Action：调用工具、输出文本、请求确认、结束

Transition：动作改变外部世界或内部状态

Reward：任务成功、用户满意、成本低、安全合规

为什么是 POMDP 而不是 MDP？
因为 Agent 从来拿不到完整世界状态，它只能看到局部信息。
例如“帮我整理客户邮件并更新 CRM”这个任务里：

客户是不是很生气？未必直接可见

CRM 数据是否最新？未必知道

邮件线索是否完整？未必

用户真正想优先处理谁？未必清楚

所以它只能在不完全信息下行动。
这就是很多 Agent 设计最终会落到：

记忆

检索

反思

澄清

计划修正

这些机制本质上都是在弥补部分可观测问题。

六、那 LLM 在 Agent 里更像 RL 里的什么？
这个问题很重要。
LLM 在 Agent 系统中可能扮演很多角色，而不仅是“policy”。
角色 1：策略函数
最直接
输入当前上下文，输出下一步动作：

回复用户

调工具

调哪个工具

参数是什么

这相当于 policy。

角色 2：价值评估器
例如让模型判断：

这个计划靠谱不靠谱？

当前结果是否足够完成任务？

哪条搜索结果最有希望？

这个候选答案哪个更好？

这有点像 value function 或 critic。

角色 3：世界模型
模型可以预测：

如果我调用这个工具，大概会返回什么

如果我发这封邮件，可能会有什么后果

这个计划执行到第三步可能卡住

这有点像 model-based RL 中的 environment model。

角色 4：规划器
模型先生成计划，再执行：

Step 1...

Step 2...

Step 3...

这有点像 classical planning + model-based search。

角色 5：状态压缩器
模型把长历史总结成短摘要，供后续使用。
这相当于 belief state update 的一种近似实现。

所以现代 LLM Agent 很特别的一点是：

同一个神经模型，同时承担 policy、critic、world model、planner、memory compressor 等多个角色。

而在传统 RL 中，这些部件常常是分开的。
这也是为什么很多人觉得 LLM Agent 有点“模糊”——因为它不像经典 RL 那样模块边界清晰。

七、Agent 和 RL 最像的地方，不在“用不用模型”，而在“闭环决策”
很多人把 Agent 理解成：

“会调用工具的 LLM”

这个说法太浅了。
更本质的是：

Agent 是闭环控制系统。

所谓闭环，就是它不是一次性输入输出，而是：

观察

决策

执行

反馈

更新

再决策

这一点和 RL、控制论、机器人学都很像。
所以从更高层看，Agent 其实同时继承了三条传统：
1. 来自 AI 的 rational agent tradition
关注感知—决策—行动
2. 来自 RL 的 sequential decision making
关注长期收益和策略优化
3. 来自控制论 / 软件工程
关注反馈闭环、稳定性、约束、故障恢复
现代工程 Agent 本质上是这三者的融合。

八、为什么今天的 Agent 和“经典 RL agent”又很不一样
虽然结构相似，但现代软件 Agent 和经典 RL agent 有几个本质不同。
1. 目标常常是语言定义，不是数值奖励
例如：

“写得礼貌一点”

“优先处理重要客户”

“不要误发”

“必要时请求审批”

这些都很难严格数值化。
所以 Agent 更多依赖：

prompt

rubric

verifier

policy constraints

human-in-the-loop

而不是一个单一 reward function。

2. 动作是符号化、组合化的
传统 RL 动作往往低层：

移动

施力

控制角度

现代 Agent 动作往往高层：

调“search_web(query=...)”

调“send_email(to, body)”

调“update_crm(customer_id, fields)”

这意味着动作空间是结构化 API 空间，不是简单向量动作空间。

3. 环境不是稳定模拟器
Atari、MuJoCo、围棋都比较封闭。
而真实 Agent 的环境是：

网页

数据库

用户

第三方系统

人工审批流程

这些环境：

非平稳

有延迟

有错误

有权限边界

有外部副作用

所以经典 RL 的许多假设并不成立。

4. 成本和安全约束极强
RL 常假设可以大量探索。
但真实 Agent 不行：

不能乱发邮件试试看

不能随便改 CRM

不能拿真实用户数据做高风险探索

不能为了探索而乱下单

因此真实 Agent 往往是：

弱探索、强约束、强审计

这和很多 RL 教科书场景差别很大。

九、从理论上说，Agent 工程里的几个热门机制，都能在 RL 里找到影子
这是最有意思的部分。
1. Planning（规划）
Agent 常见做法：

先出计划

再逐步执行

遇到错误重规划

对应 RL 里：

model-based RL

tree search

receding horizon control

hierarchical planning

2. Reflection / Self-correction（反思纠错）
Agent 常做：

检查刚才的输出

判断是否达标

不达标则重试

对应 RL 里类似于：

critic 评估

policy improvement

value-guided refinement

当然实现上不一样，但思想相通：
先行动，再评估，再改进。

3. Memory（记忆）
Agent 的记忆：

会话记忆

长期偏好

任务中间结果

检索式知识库

RL 对应：

recurrent policy

belief state

episodic memory

external memory architectures

4. Hierarchical decomposition（分层）
Agent 会：

主任务 → 子任务 → 子子任务

主 Agent → 子 Agent

RL 对应：

Hierarchical RL

options framework

skills / macro-actions

这里甚至“skill”这个词都直接重叠。

5. Tool use（工具使用）
Agent 里动作是工具调用。
RL 里可以把工具调用视为高层 action / option。
比如：

search_web 是一个 option

send_email 是一个 option

ask_human 是一个 safe fallback option

这就是典型的 hierarchical/semi-MDP 思路。

十、真正的分界线：RL 研究“最优性”，Agent 工程研究“可用性”
这是两者最深层的文化差异。
RL 的核心问题
在形式化环境中，如何学到最优或近似最优策略？
关注：

sample efficiency

regret

convergence

exploration

asymptotic performance

Agent 工程的核心问题
在真实系统中，如何做出一个能跑、靠谱、可控、可审计、成本可接受的系统？
关注：

工具调用成功率

任务完成率

可靠性

延迟

token 成本

安全

权限控制

人工兜底

可观测性和日志

所以在工程里，一个“好 Agent”未必是最优策略，只要它：

够稳

够便宜

足够好

能回滚

不出大事故

就已经很优秀了。
这和 RL 的“最大化期望回报”取向不太一样。

十一、你可以这样理解二者层级
一个比较清晰的层级图是：
第 1 层：Agent 抽象
任何能感知、决策、行动的系统

所以：
RL 主要覆盖第 2～3 层里的“学习策略”部分；现代 Agent 工程横跨第 1～4 层。

十二、一个非常实用的判断标准：什么时候问题更像“Agent”，什么时候更像“RL”？
更像 Agent 的情况
当你更关心：

如何把任务做完

如何调工具

如何维护状态

如何加审批和权限

如何设计运行时循环

如何防止错误外溢

例如：

邮件助手

销售自动化

研究助理

运维自动修复

客服工作流

更像 RL 的情况
当你更关心：

如何通过反馈学习策略

如何处理长期回报

如何探索未知动作

如何优化交互序列

如何做 credit assignment

例如：

机器人控制

游戏策略学习

推荐系统长期优化

动态定价

在线广告

连续交互中的行为优化

两者结合的情况
当你已经有一个 Agent runtime，但想优化它的决策策略：

什么时候该搜，什么时候该直接答

哪个工具优先级最高

什么时候该请求人工确认

多步任务怎么分解最有效

怎么根据长期用户反馈改进

这时 RL 就可能进入：

tool selection policy

stopping policy

clarification policy

budget allocation policy

routing policy

也就是说：

Agent 是壳，RL 可以用来优化壳里的决策器。

十三、你这个定义下，Agent 最接近哪种 RL 数学对象？

十四、为什么很多人讨论 Agent 时容易和 RL 混淆
因为“Agent”这个词在不同社区里意思不同。
在 RL / 机器人学里
Agent 往往指：

在环境中采取动作并接收奖励的决策体

在 LLM / AI 产品工程里
Agent 往往指：

能围绕目标自主调用工具并循环执行的应用系统

前者默认带有决策理论和学习问题。
后者默认带有工作流、软件集成和运行时问题。
所以混淆是正常的，本来就是两个传统汇流后的同名概念。

十五、最本质的一句话总结
你可以把两者关系概括成下面这三句：
1
Agent 是“谁在做决策和行动”的系统概念。
2
强化学习是“这个决策系统如何通过奖励信号学会更好行动”的理论与方法。
3
现代 LLM Agent 更多是“运行中的闭环软件系统”，而不是“正在做强化学习的系统”。

十六、给你一个工程上最有用的框架

二、把 LLM Agent 映射到 POMDP 的每个部分
先把你熟悉的工程对象一一塞进去。
1. 隐状态 SSS：真实世界全状态
这不是模型看到的状态，而是“世界真实是什么样”。
对一个邮件 + CRM Agent，隐状态可能包括：

用户真实意图

当前所有邮件完整内容

CRM 当前数据库真实值

外部 API 的真实状态

是否存在未显式展示的约束

任务当前是否已被某个外部流程部分执行

组织权限状态

真实的业务环境上下文

这些信息 Agent 通常拿不全。
所以它面临的是部分可观测环境。

2. 观测 OOO：Agent 每轮能看到什么
LLM Agent 的观测通常是异构的：

用户消息

系统提示

工具返回

检索片段

网页文本

数据库查询结果

错误码

审批结果

历史摘要

形式上可以记为：

ot=(user_msg,tool_results,memory_retrievals,runtime_signals,… )o_t = (\\text{user\_msg}, \\text{tool\_results}, \\text{memory\_retrievals}, \\text{runtime\_signals}, \\dots)ot​=(user_msg,tool_results,memory_retrievals,runtime_signals,…)
所以 LLM Agent 的 observation space 往往不是单一向量，而是多模态/结构化观测空间。

3. 动作 AAA：不仅是输出文本，而是完整动作集合
LLM Agent 的动作通常至少有三类：
纯语言动作

回复用户

请求澄清

解释结果

输出最终答案

工具动作

search_web(q)

query_db(sql)

send_email(to, body)

update_crm(id, fields)

call_subagent(task_spec)

控制动作

terminate

at=(τt,θt)a_t = (\\tau_t, \\theta_t)at​=(τt​,θt​)

这已经不是简单离散动作，而是参数化动作空间。

4. 转移函数 TTT：动作如何改变世界
T(st+1∣st,at)T(s_{t+1} \\mid s_t, a_t)T(st+1​∣st​,at​)
在 LLM Agent 里，这个转移比经典 RL 更复杂，因为有些动作只改变内部状态，有些动作改变外部世界。
例如：

回复用户：主要改变对话状态

搜索网页：外部世界不变，但 Agent 获得新信息

发邮件：外部世界发生真实副作用

更新 CRM：数据库状态真的变了

请求审批：未来可能产生异步事件

st+1=fenv(st,at)s_{t+1} = f_{\\text{env}}(s_t, a_t)st+1​=fenv​(st​,at​)

mt+1=fmem(mt,ot+1,at)m_{t+1} = f_{\\text{mem}}(m_t, o_{t+1}, a_t)mt+1​=fmem​(mt​,ot+1​,at​)
这点非常关键：
经典 POMDP 只有环境状态；工程 Agent 还需要显式内部工作记忆。

5. 观测模型 Ω\\OmegaΩ：为什么工具返回也是 observation
Ω(ot+1∣st+1,at)\\Omega(o_{t+1} \\mid s_{t+1}, a_t)Ω(ot+1​∣st+1​,at​)
比如你执行 query_db(...)，真正发生的是：

环境已经有了数据库真实状态 st+1s_{t+1}st+1​

系统给你返回一个结果 ot+1o_{t+1}ot+1​

这个结果可能：

不完整

有截断

有延迟

有权限过滤

有报错

所以工具输出天然就是 observation，而不是 state 本身。
这解释了一个很重要的问题：

工具调用不是“直接看到真相”，而是“通过一个观测通道获取关于真相的部分证据”。

这正是 POMDP 视角。

6. 奖励 RRR：在 LLM Agent 里往往是隐式或后验定义
标准 RL 要写：
R(st,at,st+1)R(s_t, a_t, s_{t+1})R(st​,at​,st+1​)
但现实 Agent 很多时候没有显式奖励。
所以通常有三种处理法：
方式 A：显式奖励
比如：

任务完成 +10

发错邮件 -100

每次工具调用 -0.1

请求人工审批 -0.2

超预算 -5

方式 B：稀疏终局奖励

成功完成任务 +1

失败 0 或 -1

方式 C：把 reward 换成约束优化目标
更符合工程 reality：

成功率最大化

风险不超过阈值

token 成本最小

延迟受限

必须满足审批规则

这时比起标准 POMDP，更像 Constrained POMDP。

三、为什么“普通 POMDP”还不够
如果只说“LLM Agent 是 POMDP”，是对的，但还不够严格，因为少了三个现实关键点：

它有内部记忆

它的动作往往是持续时间不等的宏动作

它的动作带参数且可能调用外部子过程

所以我们要继续升级。

四、引入内部记忆：把 Agent 写成 belief-state controller 或 memory-augmented controller
LLM Agent 绝不是无状态 policy。
它需要维护：

对话历史摘要

当前计划

已完成步骤

工具返回缓存

可用预算

错误恢复计数

用户偏好

长期记忆检索结果

因此，一个更好的形式化是：
内部记忆状态
mt∈Mm_t \\in \\mathcal{M}mt​∈M
记忆更新函数
mt+1=U(mt,ot+1,at)m_{t+1} = U(m_t, o_{t+1}, a_t)mt+1​=U(mt​,ot+1​,at​)
策略
at∼π(a∣mt)a_t \\sim \\pi(a \\mid m_t)at​∼π(a∣mt​)

五、为什么要上升到 Semi-MDP
现在来到最关键的一步。
普通 POMDP/MDP 默认每一步动作的时间粒度差不多，而且一步动作一步转移。
但 Agent 不是这样。
例如这些动作：

m_{t+1}=U(m_t,o_{t+1},u_t)ut​∼Πθ​(⋅∣mt​),mt+1​=U(mt​,ot+1​,ut​)
这里没有独立的显式 plan variable。

ReAct 的特点
它其实最接近：

POMDP 中的 receding horizon control

或者“每步重算”的 online policy

它不提前求一条完整路径，而是：

看一步，走一步，依据反馈实时调整。

ReAct 的优缺点在数学上对应什么
优点

对观测误差、环境变化更鲁棒

不要求提前获得准确任务分解

能在部分可观测环境中持续更新 belief/memory

代价

长程一致性弱

容易局部贪心

容易出现 action oscillation

由于没有稳定高层 plan，轨迹方差较大

从控制角度说，ReAct 更像反馈强、前瞻弱。

三、Plan-and-Execute：显式把 policy 拆成高层规划器和低层执行器
Plan-and-Execute 和 ReAct 最大区别，不是“先想一下”，而是：

它把计划本身变成了内部显式状态变量，并把决策分成两种时间尺度。

典型流程：

先生成 plan：p=(g1,g2,…,gk)p = (g_1, g_2, \\dots, g_k)p=(g1​,g2​,…,gk​)

再执行每个子步骤

必要时修订计划

1. 加一个 plan 变量
现在内部状态不只是 history，而是：
mt=(zt,pt,it)m_t = (z_t, p_t, i_t)mt​=(zt​,pt​,it​)
其中：

ztz_tzt​：普通上下文/记忆

ptp_tpt​：当前计划

iti_tit​：当前执行到计划第几步

2. 分成两个策略
高层规划策略
pt∼Πplan(⋅∣zt)p_t \\sim \\Pi_{\\text{plan}}(\\cdot \\mid z_t)pt​∼Πplan​(⋅∣zt​)
低层执行策略
ut∼Πexec(⋅∣zt,pt,it)u_t \\sim \\Pi_{\\text{exec}}(\\cdot \\mid z_t, p_t, i_t)ut​∼Πexec​(⋅∣zt​,pt​,it​)
这就是标准分层控制。

3. 它更接近 hierarchical POMDP / options
可以把每个计划步骤 gig_igi​ 看成一个 subgoal 或 option。
那么整个系统变成：

高层：选择/更新 option 序列

低层：在当前 option 下采取行动

形式上像：
gk∼Πhigh(⋅∣zk)g_k \\sim \\Pi_{\\text{high}}(\\cdot \\mid z_k)gk​∼Πhigh​(⋅∣zk​)
ut∼Πlow(⋅∣zt,gk)u_t \\sim \\Pi_{\\text{low}}(\\cdot \\mid z_t, g_k)ut​∼Πlow​(⋅∣zt​,gk​)
直到当前 gkg_kgk​ 终止，再切到下一项。

Plan-and-Execute 和 ReAct 的本质差异
最核心的数学差异是：
ReAct
ut∼Π(u∣mt)u_t \\sim \\Pi(u \\mid m_t)ut​∼Π(u∣mt​)
只有一个在线策略。
Plan-and-Execute
p∼Πplan(p∣z),ut∼Πexec(u∣zt,p,it)p \\sim \\Pi_{\\text{plan}}(p \\mid z), \\qquad
u_t \\sim \\Pi_{\\text{exec}}(u \\mid z_t, p, i_t)p∼Πplan​(p∣z),ut​∼Πexec​(u∣zt​,p,it​)
它把计划变量显式化了。

这意味着什么
这意味着 Plan-and-Execute 比 ReAct 多了一个约束源：

ReAct 每步只受当前历史约束

Plan-and-Execute 每步还受“当前计划”约束

所以执行时的动作分布不再是仅由当前 observation 决定，而是由 observation + plan 共同决定：
Πexec(ut∣zt,pt,it)\\Pi_{\\text{exec}}(u_t \\mid z_t, p_t, i_t)Πexec​(ut​∣zt​,pt​,it​)
这会带来：
好处

长程一致性更强

对复杂任务可分解性更好

更易审计和解释

坏处

计划错误会系统性传播

如果环境变化快，旧计划会变成负担

需要 plan revision 机制，否则刚性过强

所以数学上它是分层、低频规划、高频执行。

四、Toolformer：把“是否调用工具”内化到 token 生成策略中
Toolformer 最容易被误解，因为它不是主要在讲 runtime loop，而是在讲：

模型参数化策略如何学会在生成过程中插入工具调用。

所以它和 ReAct、Plan-and-Execute 不在同一层面上。
ReAct/Plan-and-Execute 主要是运行时控制结构。
Toolformer 主要是策略表示与训练方式。

1. Toolformer 的核心形式
普通语言模型是：
pθ(xt∣x<t)p_\\theta(x_t \\mid x_{<t})pθ​(xt​∣x<t​)
Toolformer 扩展为可以生成特殊 token / API call span：
pθ(yt∣y<t)p_\\theta(y_t \\mid y_{<t})pθ​(yt​∣y<t​)
其中 yty_tyt​ 既可以是普通文本 token，也可以是：

工具调用起始标记

工具名

参数

工具结果插入位

所以动作选择被细化到 token 序列内部。

2. 从 Agent 角度看，Toolformer 做了什么
它把原来显式的高层动作：
ut∈{search,calc,lookup,respond}u_t \\in \\{\\texttt{search}, \\texttt{calc}, \\texttt{lookup}, \\texttt{respond}\\}ut​∈{search,calc,lookup,respond}
变成了语言生成过程中的一种特殊结构化输出模式。
即：

原来：runtime 在一步中决定 “call tool”

Toolformer：LM 在生成序列时自己长出工具调用片段

所以更像：
Πθ(next token∣context)\\Pi_\\theta(\\text{next token} \\mid \\text{context})Πθ​(next token∣context)
而不是显式：
Πθ(next action∣mt)\\Pi_\\theta(\\text{next action} \\mid m_t)Πθ​(next action∣mt​)

3. 数学上它和 ReAct 差在哪
ReAct 的动作空间是显式高层动作：
UReAct=Utext∪Utool∪Ucontrol\\mathcal{U}_{\\text{ReAct}} = \\mathcal{U}_{text} \\cup \\mathcal{U}_{tool} \\cup \\mathcal{U}_{control}UReAct​=Utext​∪Utool​∪Ucontrol​
Toolformer 则更像把工具调用编码进扩展词表或结构化 token 序列：
Y=Vtext∪Vtool_syntax\\mathcal{Y} = \\mathcal{V}_{text} \\cup \\mathcal{V}_{tool\_syntax}Y=Vtext​∪Vtool_syntax​
因此：

ReAct：action-level decision making

Toolformer：token-level latent action emission

4. 更严格地说
Toolformer 可以被看作一种把 option invocation 编码为可学习序列决策模式的方法。
也就是原来你有一个外显动作：
ut=call_tool(name,args)u_t = \\texttt{call\_tool}(name,args)ut​=call_tool(name,args)
现在变成模型生成：
[<API> name args </API>][\\texttt{<API>} \\ \\texttt{name} \\ \\texttt{args} \\ \\texttt{</API>}][<API> name args </API>]
然后外部 runtime 执行并把结果插回来。
所以它对应的是：

策略参数化方式改变了，但环境和工具仍然存在。

5. Toolformer 的特殊点
它把“工具调用时机”学习成了局部生成决策，而不是显式 planner 决策。
这意味着它更适合：

局部可触发、低延迟工具

工具使用模式相对稳定

调用逻辑可从数据中模仿学习

但不擅长：

长程任务分解

复杂计划维护

强约束执行流

所以从框架层级看：

ReAct / Plan-and-Execute：偏 runtime architecture

Toolformer：偏 policy representation/training

五、Multi-Agent：把单一控制器拆成多个局部控制器
Multi-Agent 也容易被说得太玄。其实数学上最关键的一点是：

单 Agent 有一个策略 Π\\PiΠ，Multi-Agent 有多个局部策略 Π1,Π2,…,Πn\\Pi_1,\\Pi_2,\\dots,\\Pi_nΠ1​,Π2​,…,Πn​。

然后问题变成：

它们共享不共享状态？

它们如何通信？

谁协调？

是否有全局奖励？

1. 最简单的 Multi-Agent 形式
定义多个 Agent：
Gi=(S,Mi,Ui,Oi,Ui,Πi)\\mathcal{G}_i = (\\mathcal{S}, \\mathcal{M}_i, \\mathcal{U}_i, \\mathcal{O}_i, U_i, \\Pi_i)Gi​=(S,Mi​,Ui​,Oi​,Ui​,Πi​)
每个 Agent 有：

自己的局部记忆 mim_imi​

自己的动作空间 Ui\\mathcal{U}_iUi​

自己的观测 Oi\\mathcal{O}_iOi​

环境是共享的，但每个 Agent 看到的只是局部观测。
这就是典型 Dec-POMDP 味道：
ut(i)∼Πi(⋅∣mt(i))u_t^{(i)} \\sim \\Pi_i(\\cdot \\mid m_t^{(i)})ut(i)​∼Πi​(⋅∣mt(i)​)
联合动作：
ut=(ut(1),…,ut(n))\\mathbf{u}_t = (u_t^{(1)}, \\dots, u_t^{(n)})ut​=(ut(1)​,…,ut(n)​)
共同影响环境：
st+1∼T(st,ut)s_{t+1} \\sim T(s_t, \\mathbf{u}_t)st+1​∼T(st​,ut​)

2. 但多数 LLM Multi-Agent 不是对称 Dec-POMDP
真实系统里更常见的是不对称结构：

Planner Agent

Research Agent

Critic Agent

Executor Agent

Safety Agent

这不是平权多智能体，而是功能分工的分层/模块化控制。
所以多数 Multi-Agent LLM 系统更准确地说是：

由多个专门化 option-controller 组成的组织化系统

而不是经典博弈论里的平等多主体系统。

3. Multi-Agent 的几种数学结构
结构 A：分工式串行委托
主 Agent 选择子 Agent 执行子任务：
it∼Πrouter(⋅∣mt)i_t \\sim \\Pi_{\\text{router}}(\\cdot \\mid m_t)it​∼Πrouter​(⋅∣mt​)
ut∼Πi(⋅∣mt(i))u_t \\sim \\Pi_i(\\cdot \\mid m_t^{(i)})ut​∼Πi​(⋅∣mt(i)​)
这更像mixture-of-experts + options。
结构 B：并行协作
多个 Agent 同时对同一问题提建议，再汇总：
ut(i)∼Πi(⋅∣mt(i))u_t^{(i)} \\sim \\Pi_i(\\cdot \\mid m_t^{(i)})ut(i)​∼Πi​(⋅∣mt(i)​)
ut=A(ut(1),…,ut(n))u_t = A(u_t^{(1)}, \\dots, u_t^{(n)})ut​=A(ut(1)​,…,ut(n)​)
这里 AAA 是聚合器。
结构 C：辩论/评审
一个生成，一个批评，一个裁决：
y(1)∼Πpropose(⋅)y^{(1)} \\sim \\Pi_{\\text{propose}}(\\cdot)y(1)∼Πpropose​(⋅)
c(2)∼Πcritic(⋅∣y(1))c^{(2)} \\sim \\Pi_{\\text{critic}}(\\cdot \\mid y^{(1)})c(2)∼Πcritic​(⋅∣y(1))
y∗∼Πjudge(⋅∣y(1),c(2))y^* \\sim \\Pi_{\\text{judge}}(\\cdot \\mid y^{(1)}, c^{(2)})y∗∼Πjudge​(⋅∣y(1),c(2))
这本质上是把单模型内部 self-reflection 外化成多个显式 policy。

4. Multi-Agent 和单 Agent 最大数学差别
不是“人数多了”，而是：
单 Agent
ut∼Π(⋅∣mt)u_t \\sim \\Pi(\\cdot \\mid m_t)ut​∼Π(⋅∣mt​)
Multi-Agent
ut(i)∼Πi(⋅∣mt(i),ct(i))u_t^{(i)} \\sim \\Pi_i(\\cdot \\mid m_t^{(i)}, c_t^{(i)})ut(i)​∼Πi​(⋅∣mt(i)​,ct(i)​)
ut=A({ut(i)}i=1n)u_t = A(\\{u_t^{(i)}\\}_{i=1}^n)ut​=A({ut(i)​}i=1n​)
其中 ct(i)c_t^{(i)}ct(i)​ 是通信上下文，AAA 是协调/聚合机制。
所以 Multi-Agent 多了两个对象：

通信协议

协调算子

这两个在单 Agent 里通常是隐含在一个大 prompt 中的。

六、四者放在一起，核心差异到底在哪
下面直接按“数学结构差异”列出来。

1. ReAct：策略是否显式分层？
没有。
ut∼Π(u∣mt)u_t \\sim \\Pi(u \\mid m_t)ut​∼Π(u∣mt​)
它是一层 policy，每一步根据最新记忆决定下一步。
本质

单层 policy

在线反馈控制

不显式维护长期计划变量

2. Plan-and-Execute：是否引入显式计划变量？
有。
p∼Πplan(p∣z)p \\sim \\Pi_{\\text{plan}}(p \\mid z)p∼Πplan​(p∣z)
ut∼Πexec(u∣zt,p,it)u_t \\sim \\Pi_{\\text{exec}}(u \\mid z_t, p, i_t)ut​∼Πexec​(u∣zt​,p,it​)
本质

两层 policy

plan 是显式 latent variable

行动受 plan 条件化

3. Toolformer：工具调用是在 action level 还是 token level？
在 token level 内化。
yt∼pθ(yt∣y<t)y_t \\sim p_\\theta(y_t \\mid y_{<t})yt​∼pθ​(yt​∣y<t​)
其中 yty_tyt​ 可包含工具调用语法。
本质

策略表示变化

工具调用被编进生成模型

更像 learned API-invocation language model

4. Multi-Agent：是一个 policy 还是多个相互作用的 policy？
多个。
ut(i)∼Πi(⋅∣mt(i))u_t^{(i)} \\sim \\Pi_i(\\cdot \\mid m_t^{(i)})ut(i)​∼Πi​(⋅∣mt(i)​)
ut=A(ut(1),…,ut(n))u_t = A(u_t^{(1)},\\dots,u_t^{(n)})ut​=A(ut(1)​,…,ut(n)​)
本质

分布式策略系统

多局部记忆

多动作源 + 协调机制

七、从“latent variable”角度看更清楚
这是个很干净的视角。
ReAct
latent 很少，主要是当前 scratchpad / memory：
mtm_tmt​
Plan-and-Execute
多了显式 latent plan：
ptp_tpt​
Toolformer
把工具调用边界和参数嵌进生成序列里的 latent structure：
y1:Ty_{1:T}y1:T​
Multi-Agent
多了 agent index、通信状态、局部信念等 latent：
(mt(1),…,mt(n),ct)(m_t^{(1)}, \\dots, m_t^{(n)}, c_t)(mt(1)​,…,mt(n)​,ct​)
所以可以说，这四种方法的根本差别在于：

它们引入了不同的隐变量结构来约束决策过程。

八、从控制频率看，也能看出差异
ReAct
高频闭环
每一步都重新决策。
Plan-and-Execute
低频规划 + 高频执行
计划更新频率低于动作更新频率。
Toolformer
最高频局部决策
甚至 token 级决定是否调用工具。
Multi-Agent
多时钟系统
不同 Agent 可有不同更新频率和不同控制粒度。
这是个很重要的工程差别。

九、从误差传播方式看，四者也不同
ReAct
误差局部传播。
因为每一步都能纠偏，但也容易局部抖动。
Plan-and-Execute
误差全局传播。
高层 plan 错了，下面全受影响。
Toolformer
误差是局部触发错误。
比如调用时机错、参数错、结果融入文本错。
Multi-Agent
误差可能来自协调失败。
不是单点错误，而是通信/委托/聚合链路错误。
这就是为什么 Multi-Agent 往往不是更稳，而是更复杂。

十、从 RL / HRL 的对应关系看
ReAct
最接近：

online POMDP policy

Plan-and-Execute
最接近：

hierarchical RL

Toolformer
最接近：

十一、它们不是互斥类别，而是可叠加的
这一点很关键。
现实系统常常是混合的：
例子 1
一个 Multi-Agent 系统里，每个子 Agent 都用 ReAct。
例子 2
一个 Plan-and-Execute Agent，其 executor 用 Toolformer 风格内化工具调用。
例子 3
一个 Planner Agent 负责任务分解，多个 specialist agent 并行执行，每个 specialist 内部 ReAct。
所以这几类更像是在不同维度上的设计选择：

ReAct：runtime control style

Plan-and-Execute：hierarchy structure

Toolformer：policy parameterization/training style

Multi-Agent：controller decomposition style

不是同一维度上的四选一。

十四、最后给一个判断口诀
以后看到一个新 Agent 框架，你可以直接问四个问题：

它有没有显式 plan 变量？
有，就是 Plan-and-Execute 味。

工具调用是在高层动作空间里，还是被内化到 token 生成里？
后者就是 Toolformer 味。

系统里是一个 policy 还是多个 policy 在协同？
多个就是 Multi-Agent 味。

它是每一步依据反馈重决策，还是按预定计划推进？
前者偏 ReAct，后者偏 Plan-and-Execute。
